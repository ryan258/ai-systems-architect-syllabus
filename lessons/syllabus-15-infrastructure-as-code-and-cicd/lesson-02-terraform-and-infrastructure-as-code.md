# Lesson 15.02: Terraform and Infrastructure as Code

**Parent syllabus:** [Syllabus 15: Infrastructure as Code and CI/CD](../../syllabus-15-infrastructure-as-code-and-cicd.md)  
**Estimated time:** 2-3 hours  
**Artifact:** Terraform stack sketch and state-handling policy

---

## Outcome

By the end of this lesson, you can sketch a minimum useful Terraform setup for a small AI service, explain how plans and state work, and document how infrastructure changes are reviewed and recovered.

You will produce:

- a Terraform stack sketch
- a state-handling policy
- an infrastructure review checklist

---

## Why This Matters

Dashboard clicking does not scale past the first deployment.

Common failures:

- Cloud resources exist, but nobody can explain how to recreate them from source.
- Infrastructure changes are applied directly in production without review.
- Terraform state lives on one laptop and becomes a hidden single point of failure.
- Secrets leak into variables, plans, or state snapshots.
- Manual dashboard changes create drift that later breaks applies.
- Cost-bearing resources are created casually because the change is never reviewed as code.

Infrastructure as code is not about abstraction for its own sake. It is about making runtime shape reviewable, repeatable, and recoverable.

---

## Best-Practice Principles

1. **Start with the minimum useful stack.**  
   Small systems usually need clear providers, a handful of resources, remote state, and environment-specific inputs, not an elaborate module forest.

2. **Treat `plan` as a review artifact.**  
   Operators should see what infrastructure is expected to change before `apply`.

3. **Protect state like production metadata.**  
   State often reveals resource identifiers, config values, and sensitive relationships.

4. **Prefer explicit ownership and naming over clever abstraction.**  
   Small teams benefit more from clear files than from premature generalization.

5. **Assume drift will happen and define how to handle it.**  
   The question is not whether drift appears. The question is whether you notice and recover safely.

6. **Keep secrets out of code and minimize them in state.**  
   Reference secret managers where possible instead of embedding raw values.

7. **Make cost visible in infrastructure review.**  
   Capacity, storage, and managed services should be examined before merge, not after the bill arrives.

## Concepts

### Infrastructure as Code

Declaring infrastructure in version-controlled files so desired state can be reviewed, applied, and recreated.

### Provider

The Terraform plugin that knows how to manage resources for a given platform or service.

### Resource

A concrete infrastructure object Terraform manages.

Examples:

- service account
- container service
- secret reference
- storage bucket

### Variable

An input that lets the same configuration work across environments without duplicating the entire stack.

### State

Terraform's record of what it believes exists and how managed objects map to real infrastructure.

### Drift

A mismatch between declared infrastructure and the actual deployed environment.

## Walkthrough

Use this example:

> A support drafting API now runs in the cloud and needs a repeatable deployment foundation for staging and production.

### Step 1 - Keep the first stack small

For a small service, the first Terraform sketch might manage:

- the container runtime or service definition
- service identity and access policy
- secret references
- network or ingress settings
- logging or monitoring hooks

It might explicitly leave out:

- complex shared networking owned elsewhere
- provider-unsupported resources
- temporary demo infrastructure

The point is to codify the service boundary that must be repeatable.

### Step 2 - Choose a simple repository layout

A minimum layout is usually enough:

```text
infra/
  versions.tf
  providers.tf
  variables.tf
  main.tf
  outputs.tf
  env/
    staging.tfvars
    production.tfvars
```

That gives you:

- version pinning
- provider declaration
- environment input separation
- one obvious place to review changes

### Step 3 - Write the state policy early

For this service:

- state should live in a managed remote backend
- backend storage should be encrypted and versioned
- apply access should be limited to named operators or CI identities
- local state files should not be committed
- state recovery steps should be written before the first incident

State policy is operational policy. Do not leave it implied.

### Step 4 - Use plan review as a gate

Before applying infrastructure changes:

1. Generate a plan for the target environment.
2. Review the exact creates, updates, and destroys.
3. Check for privacy, security, cost, and downtime implications.
4. Approve only after the blast radius is understood.

For AI systems, this matters because infrastructure changes often affect:

- network exposure
- secret access
- scaling cost
- observability coverage
- dependency permissions

### Step 5 - Decide how drift will be handled

For this workflow, drift might come from:

- an operator changing runtime settings in the dashboard
- an emergency secret reference update
- a manually added alert or IAM binding

A practical drift rule could be:

- run a scheduled plan or drift check weekly and before production releases
- block release if unmanaged drift touches security, scaling, secret access, or ingress
- reconcile drift back into code before further expansion

### Step 6 - Keep secrets out of Terraform where possible

Safer patterns:

- Terraform creates secret references, not raw secret values
- CI injects identifiers or bindings, not the secret content
- environment variables point to secret manager entries

Less safe pattern:

- raw secrets passed through tfvars, plan output, and state

That pattern creates privacy and governance problems that are hard to unwind later.

## Practice

Choose one deployable AI service and draft a **Terraform Stack Sketch** that includes:

1. repository layout
2. managed resource inventory
3. state backend and recovery rules
4. plan review flow
5. drift detection policy
6. cost and security review notes

Keep the stack intentionally small. If you are unsure whether a resource belongs in Terraform yet, state why.

---

## Review Checklist

- The stack scope is explicit.
- Managed resources are named with purpose, not just type.
- Environment differences are handled through inputs, not duplicate stacks.
- State storage, access, backup, and recovery are defined.
- Plan review is part of the normal workflow.
- Secret handling avoids raw secret values where possible.
- Drift detection and reconciliation rules are documented.
- Cost, observability, and access risks are visible in review.

---

## Common Failure Modes

- Writing Terraform files without deciding where state lives.
- Treating `apply` as a personal admin action instead of a reviewed change.
- Overengineering modules before the first stack is stable.
- Managing raw secrets directly in Terraform inputs.
- Ignoring provider version pinning and getting surprise behavior changes.
- Allowing dashboard drift to accumulate until the next apply becomes risky.
- Expanding the stack scope faster than the team can operate it.
