# Lesson 04.04: Building a Completion Habit

**Parent syllabus:** [Syllabus 04: Finishing Systems](../../syllabus-04-finishing-systems.md)  
**Estimated time:** 2 hours  
**Artifact:** Weekly completion review system

---

## Outcome

By the end of this lesson, you can run a weekly review that closes, pauses, reduces, or advances open projects.

You will produce a weekly completion review system with decision rules and a low-energy operating mode.

---

## Why This Matters

Open loops multiply when there is no review system.

For AI systems work, the problem is not just personal organization. Open loops can become stale prompts, unreviewed risks, undocumented decisions, forgotten client follow-ups, or unfinished portfolio proof.

Common failures:

- Projects stay open because there is no decision ritual.
- New ideas get more attention than old commitments.
- The review process is too large for low-energy days.
- The project list mixes active, paused, abandoned, and someday work.
- Small closure actions are delayed until they feel large.
- There is no visible record of what shipped.

A completion habit is not about forcing constant productivity. It is a system for keeping commitments honest.

---

## Best-Practice Principles

1. **Review open loops on a fixed cadence.**  
   Weekly is usually frequent enough to prevent drift.

2. **Use status labels.**  
   Active, finish this week, pause, abandon, someday, and shipped should mean different things.

3. **Limit work in progress.**  
   Too many active projects make finishing harder.

4. **Make closure actions small.**  
   A two-minute action should be done immediately during review.

5. **Design for variable energy.**  
   The review should have a low-energy version that still works.

6. **Write decisions down.**  
   A paused project should have a reason and resume trigger.

7. **Track shipped artifacts.**  
   Completion becomes real when there is a record of what was delivered.

## Concepts

### Weekly Completion Review

A recurring review of open projects focused on decisions and closure.

### Work in Progress Limit

A cap on how many projects can be active at once.

Example:

> No more than two active build projects and one writing project.

### Two-Minute Closure

A closure action that takes less than two minutes and should be done immediately.

Examples:

- Move file to final folder
- Add link to README
- Send final note
- Rename draft
- Mark project paused
- Delete obsolete idea

### Resume Trigger

The condition that makes a paused project active again.

Example:

> Resume after client sends sample data.

## Walkthrough

Unstructured project list:

- Proposal workflow
- Better website
- AI architecture notes
- Client onboarding template
- YouTube idea
- Prompt library
- Draft case study
- Old automation experiment

This list does not tell you what to finish.

### Weekly review version

| Project | Status | Decision | Next action |
| --- | --- | --- | --- |
| Client onboarding template | Finish this week | MVD only | Add scope clauses |
| Proposal workflow | Active | Continue | Write done condition |
| Draft case study | Pause | Needs screenshot | Resume when screenshot exists |
| Old automation experiment | Abandon | No current value | Archive notes |
| Website | Someday | Too broad now | Keep idea only |
| Prompt library | Shipped | Done | Add to portfolio log |

### Low-energy version

If energy is low, answer only four questions:

1. What is one thing I can ship this week?
2. What open loop can I close in two minutes?
3. What should be paused on purpose?
4. What should be abandoned?

This keeps the habit alive without requiring a full planning session.

## Practice

Create your weekly completion review system.

Include:

1. Your status labels.
2. Your work in progress limit.
3. Your weekly review template.
4. Your low-energy version.
5. Your shipped artifact log format.
6. Three projects you will classify now.
7. One thing you will finish this week.

Keep the system small enough that you would still use it on a hard week.

---

## Review Checklist

Your completion system is acceptable when:

- It has a weekly cadence.
- It separates active, paused, abandoned, someday, and shipped work.
- It includes a work in progress limit.
- It has a low-energy version.
- It includes two-minute closure actions.
- It records shipped artifacts.
- It names pause reasons and resume triggers.
- It helps you decide, not just list.

---

## Common Failure Modes

- **Review as self-criticism:** The system becomes punishment instead of decision support.
- **Too many statuses:** The review becomes harder than the work.
- **No shipped log:** Finished work disappears from memory.
- **No pause criteria:** Projects sit open indefinitely.
- **No abandon option:** Old ideas keep pretending to be commitments.
- **No low-energy mode:** The system fails exactly when it is most needed.
