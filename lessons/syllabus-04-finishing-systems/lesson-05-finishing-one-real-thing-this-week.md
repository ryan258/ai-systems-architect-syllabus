# Lesson 04.05: Finishing One Real Thing This Week

**Parent syllabus:** [Syllabus 04: Finishing Systems](../../syllabus-04-finishing-systems.md)  
**Estimated time:** 2-3 hours  
**Artifact:** Two-hour finish plan and shipped artifact log entry

---

## Outcome

By the end of this lesson, you can choose one unfinished project, reduce it to a finishable version, and complete a useful deliverable within a defined time box.

You will produce a two-hour finish plan and a shipped artifact log entry.

---

## Why This Matters

Finishing only becomes real when an artifact leaves the open-loop state.

This lesson is deliberately practical. The goal is not to design a perfect productivity system. The goal is to finish one real thing and prove that your scope, done condition, and MVD habits work under time pressure.

Common failures:

- Choosing a project that is too large.
- Spending the session rethinking the whole project.
- Improving the wrong part.
- Starting from scratch instead of packaging what exists.
- Treating the time box as optional.
- Finishing privately but not saving evidence.

A finish session should be narrow, timed, and biased toward closure.

---

## Best-Practice Principles

1. **Pick the smallest valuable open loop.**  
   Do not choose the most emotionally loaded project first.

2. **Define the finish before the session starts.**  
   The timer should not begin until the done condition is written.

3. **Package before expanding.**  
   Turn what exists into something usable before adding new features.

4. **Use a hard time box.**  
   Time pressure helps expose what is essential.

5. **Keep a cut list visible.**  
   When energy drops, cut scope instead of extending indefinitely.

6. **End with evidence.**  
   A finished thing should have a file, link, note, delivery message, or portfolio entry.

7. **Record limitations honestly.**  
   A finished MVD can have known limits if those limits are named.

---

## Concepts

### Finish Session

A short, focused work block dedicated to closing one open loop.

### Time Box

A fixed time limit for the session.

In this lesson, use two hours or less.

### Ship Log

A record of what was finished, where it lives, what value it provides, and what was intentionally deferred.

### Completion Bias

The discipline of choosing actions that move the artifact toward acceptance rather than expansion.

## Walkthrough

Open project:

> Drafted prompt chain for writing project scope documents from discovery notes.

Two-hour finish plan:

### Done condition

> This is done when the prompt chain, example input, example output, review checklist, and known limitations are saved in one document that another consultant can review without needing my memory.

### Time box

Two hours.

### Work plan

| Time | Action |
| --- | --- |
| 0:00-0:15 | Gather existing prompts and notes |
| 0:15-0:45 | Clean the prompt chain |
| 0:45-1:10 | Add example input and output |
| 1:10-1:30 | Add review checklist |
| 1:30-1:45 | Add known limitations |
| 1:45-2:00 | Save, name, and log artifact |

### Cut list

- No new prompt experiments
- No UI
- No automation
- No extra templates
- No brand styling

### Ship log entry

> Finished: Scope Draft Prompt Chain v1. Value: turns discovery notes into a reviewable scope draft. Location: `/prompts/scope-draft-v1.md`. Limits: one input format, no pricing, no legal terms, human review required.

This is a finished MVD, not a perfect product.

## Practice

Choose one unfinished project.

Rules:

1. It must be finishable in two hours or less.
2. It must produce a visible artifact.
3. It must have a one-sentence done condition.
4. It must have a cut list.
5. It must end with a ship log entry.

After the session, write:

- What shipped?
- What was cut?
- What did the time box reveal?
- What would you do differently next time?

---

## Review Checklist

Your finish session is acceptable when:

- You chose one project.
- You wrote the done condition before starting.
- You used a hard time box.
- You kept a cut list.
- You produced a visible artifact.
- You saved the artifact somewhere findable.
- You wrote a ship log entry.
- You named limitations and deferred work.

---

## Common Failure Modes

- **Choosing the hardest project:** You burn the session on emotional weight instead of closure.
- **No time box:** The finish session becomes another open project.
- **New build impulse:** You add features instead of packaging.
- **No artifact location:** The finished thing cannot be found later.
- **No limitations:** The MVD looks more complete than it is.
- **No ship log:** You finish but fail to record evidence.
