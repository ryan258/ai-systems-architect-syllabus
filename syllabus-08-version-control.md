# Syllabus 08: Version Control — Organizing an Advanced Repo Library
**Status:** Hidden Gap — Organization, Not Basics
**Format:** One chat session per unit, self-paced

---

## What This Is

You already use Git fluently and have 326 public repos. You also use GitNexus for codebase mapping. This is not a beginner course. The gap here is not skill — it is organization, discoverability, and making your existing work legible to clients and collaborators.

---

## Unit 1 — Auditing 326 Repos Without Losing Your Mind

**Goal:** Build a fast system for deciding what to keep, archive, or delete.

**Topics:**
- How to bulk-audit repos using GitHub's API or CLI
- Decision criteria: active, reference, archive, delete
- How to batch-archive without losing anything important

**Session starter:**
> "Help me design a fast system for auditing 326 GitHub repos. I want decision criteria for active vs. archive vs. delete, and I want to know if there is a CLI or API method to do this at scale without clicking through each one."

---

## Unit 2 — Pinning and Profiling for Client Discovery

**Goal:** Make your GitHub profile immediately legible to a potential client who lands on it.

**Topics:**
- Pinning strategy: what six repos tell the best story about your work
- Profile README as a landing page, not a resume
- How to write repo descriptions that non-developers understand

**Session starter:**
> "Help me design a GitHub profile pinning strategy. I want the six pinned repos to tell a clear story about my AI workflow architecture work. What criteria should I use to pick them? What should the profile README say?"

---

## Unit 3 — README Standards for Your Most Important Repos

**Goal:** Define a consistent README format across your key projects.

**Topics:**
- What a client-facing README must include vs. a developer-facing one
- How to write the first three sentences so a non-technical person understands the project
- Template for AI workflow repos specifically

**Session starter:**
> "Help me define a README standard for my AI workflow repos that works for both technical collaborators and non-technical clients. What must the first three sentences always cover? Give me a template I can apply consistently."

---

## Unit 4 — Using GitNexus Effectively for Codebase Navigation

**Goal:** Get more out of the codebase mapping tool you already have.

**Topics:**
- How to use GitNexus output to write better documentation
- Using codebase maps to onboard clients or collaborators faster
- Where GitNexus falls short and how to compensate

**Session starter:**
> "Help me figure out how to use GitNexus output more effectively. I want to use it to write better documentation and onboard clients or collaborators faster. What is the best way to surface a codebase map in a README or handoff doc?"

---

## Unit 5 — Making Your Work Findable and Citable

**Goal:** Ensure your best projects show up when someone searches for your work.

**Topics:**
- GitHub topic tagging strategy
- How to write repo descriptions for search
- When to create a portfolio page that links to repos vs. relying on GitHub alone

**Session starter:**
> "Help me make my GitHub repos more discoverable. What topic tags should I use for AI workflow architecture projects? How do I write descriptions that appear in search? Should I build a separate portfolio page or rely on GitHub?"

---

## Practice Project

Audit your top 20 most-viewed or most-starred repos. Apply the README standard from Unit 3 to the top five. Add topic tags to all 20.
